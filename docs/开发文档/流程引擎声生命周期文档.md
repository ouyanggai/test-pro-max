## 1. 架构概览

### 1.1 核心架构模式：Template-Proxy（模板-代理模式）

系统采用**模板-代理分离**架构：

```
设计时 (Template)                    运行时 (Proxy)
┌─────────────────┐                ┌─────────────────┐
│  FlowTemplate    │  ──copy──>    │  FlowProxy       │
│  FlowNodeTemplate│  ──copy──>    │  FlowNodeProxy   │
│  FlowStrategyTemplate│ ──copy──> │  FlowStrategyProxy│
│  FlowNodeAuditConfigTemplate│    │  FlowNodeAuditConfigProxy│
│  FlowNodeFieldPowerTemplate│    │  FlowNodeFieldPowerProxy│
└─────────────────┘                └─────────────────┘
```

- **Template 表**：存储流程设计数据，修改不影响运行中的实例
- **Proxy 表**：首次提交时从 Template 复制，运行中的实例使用 Proxy 数据
- 支持运行时对 Proxy 进行修改（如加签操作），不影响原始模板

### 1.2 节点树结构

节点通过 `parentId` 和 `strategyId` 形成树形结构：

```
[开始节点]
    │
    ├── (strategyId) ──> [条件分支节点]
    │                       ├── (strategyId) ──> [分支A: 审批人1] ──> [审批人2]
    │                       └── (strategyId) ──> [分支B: 审批人3]
    │
    └── (parentId) ──> [普通审批节点]
```

### 1.3 数据存储

| 数据类型 | 存储位置 | 说明 |
|----------|----------|------|
| 表单数据 | MongoDB | 通过 FormDataService 存储 |
| 流程实例 | MySQL | FlowInstance 表 |
| 审批记录 | MySQL | FlowAuditRecord 表 |
| 操作日志 | MySQL | FlowInstanceOperationLog 表 |
| 附件 | 文件存储 | FlowInstanceAttachment |
| 审核锁 | Redis | FLOW_AUDIT_LOCK，TTL=3秒 |
| 用户缓存 | Redis | UserCache，key=sid |
| 模板缓存 | Redis | FLOW_TEMPLATE_DATA_{id}，TTL=120小时 |

## 2. 流程模板创建

### 2.1 三种流程类型

| 类型 | type 值 | 路由 | 说明 |
|------|---------|------|------|
| 表单流程（关联表单） | `1` | `/flowLibrary/flowDetails` | 关联已有的表单模板 |
| 无表单流程 | `2` | `/flowLibrary/noFormFlow` | 不关联表单，纯审批流 |
| 表单流程（设计表单） | `3` | `/flowLibrary/formFlow` | 在流程中直接设计表单 |

### 2.2 创建向导（以关联表单流程为例）

#### Step 1：基础信息

| 字段 | 控件 | 必填 | 说明 |
|------|------|------|------|
| 流程名称 (flowName) | 文本输入 | 是 | 流程模板名称 |
| 流程分组 (groupId) | 下拉选择 | 否 | 从 FlowGroup 列表选取 |
| 用途说明 (remark) | 文本域 | 是 | 流程用途描述 |

**API**: `POST /web/standardLibrary/getFlowGroup` 获取分组列表

#### Step 2：关联表单

| 字段 | 控件 | 说明 |
|------|------|------|
| 类型类别 (typeId) | 下拉选择 | 按 Type 筛选表单 |
| 标准类别 (standardId) | 下拉选择 | 按 Standard 筛选表单 |
| 表单名称 (name) | 文本搜索 | 模糊搜索表单 |

- 选择表单后可预览表单内容
- 每个被选中的表单默认所有字段设为 `only_read`（只读）

**API**: `POST /web/formLibrary/formList` 获取可选表单

#### Step 3：设计流程

设计线性审批节点链（简单流程只支持线性节点）：

- 可添加多个审批人节点
- 每个节点可配置：
  - 节点名称 (nodeName)：必填，最长20字
  - 有何权限 (hasAuth)：打开全屏对话框，为每个关联表单的每个字段设置「可编辑」权限

**保存 API**:
- 新建：`POST /web/flowTemplateApi/createCustomFlowTemplate`
- 编辑：`POST /web/flowTemplateApi/editCustomFlowTemplate`

### 2.3 模板管理操作

| 操作 | 说明 | API |
|------|------|-----|
| 列表查询 | 支持按名称、分组、类型、状态筛选 | `POST /web/flowTemplateApi/list` |
| 编辑 | 进入三步向导编辑模式 | `POST /web/flowTemplateApi/editCustomFlowTemplate` |
| 删除 | 逻辑删除，可选是否删除关联表单 | `POST /web/flowTemplateApi/delete` |
| 复制 | 复制现有模板 | `POST /web/flowTemplateApi/copy` |
| 启用/禁用 | 切换 flowStatus | `POST /web/flowTemplateApi/operation` |
| 配置流程回调 | 配置审批状态变化时的回调请求 | `POST /web/requestConfig/save` |

### 2.4 模板保存的后端流程

```
createCustomFlowTemplate()
  ├── 1. 保存 FlowTemplate 基础数据
  ├── 2. 处理表单数据 (handleFormData)
  │     ├── 保存 FormTemplate
  │     ├── 建立 FlowFormScopeTemplate 关联
  │     └── 构建 formFieldsTemplateVoMap
  ├── 3. 递归保存节点数据 (saveFlowNodeTemplateData)
  │     ├── 保存 FlowNodeTemplate
  │     ├── 保存 FlowNodeAuditConfig + FlowNodeAuditDetailConfig
  │     ├── 保存 FlowNodeFieldPowerTemplate
  │     ├── 对 condition/parallel 节点：保存 FlowStrategyTemplate + FlowStrategyConditionTemplate
  │     └── 递归保存子节点
  └── 4. 异步写入 Redis 缓存 (500ms 延迟)
```

## 3. 节点类型

### 3.1 完整节点类型列表

| 类型 (FlowNodeType) | 前端名称 | 图标 | 说明 | 可添加为子节点 |
|---------------------|----------|------|------|---------------|
| `start` | 起始节点 | -- | 流程起点，自动生成 | 不可添加 |
| `common` | 审批人 | el-icon-user-solid | 标准审批节点，需配置审核人 | 可添加 |
| `synergy` | 协同 | el-icon-edit-outline | 协同节点，多人协作处理 | 可添加 |
| `empty` | 空节点 | el-icon-s-flag | 占位节点，遍历时自动跳过 | 可添加 |
| `condition` | 条件分支 | el-icon-share | 根据表单数据条件选择分支 | 包含子分支 |
| `parallel` | 并行节点 | el-icon-s-operation | 多分支同时执行 | 包含子分支 |
| `condition` (手动) | 手动分支 | el-icon-s-promotion | 由用户手动选择分支 | 包含子分支 |
| `end` | 结束节点 | -- | 流程终点，自动生成 | 不可添加 |

### 3.2 节点配置属性

每个节点包含以下配置：

```javascript
{
  nodeName: "节点名称",           // 必填，最长20字
  type: "common",                // 节点类型
  selectMode: 0,                 // 选择模式
  selectRange: 0,                // 选择范围
  examineMode: 1,                // 审核模式
  // 审核配置
  flowNodeAuditConfig: {
    auditType: "assign",         // 审核类型
    type: "scramble",            // 审批方式：scramble(竞签) / countersign(会签)
    countersignNum: 1,           // 会签人数（countersign 模式）
    flowNodeDetailConfigList: [] // 审核人明细配置
  },
  // 字段权限
  flowNodeFieldPowerTemplateList: []  // 表单字段权限列表
}
```

## 4. 审核人员配置

### 4.1 八种审核类型 (AuditTypeEnum)

| auditType | 前端标签 | 说明 | 适用场景 | 限制条件 |
|-----------|----------|------|----------|----------|
| `assign` | 选择人员 | 指定具体用户作为审批人 | 固定审批人场景 | 始终可用 |
| `initiator` | 发起人自己 | 流程发起人即为审批人 | 发起人自审 | 不能在起始节点；仅竞签模式 |
| `run_node_choose` | 审批人自选 | 运行时由提交人/审批人选择下一审批人 | 灵活审批场景 | 不能在起始节点；仅竞签模式 |
| `role` | 选择角色 | 按角色分配审批人 | 按角色审批 | 始终可用 |
| `company_id` | 选择公司 | 按公司分配审批人 | 跨公司审批 | 仅起始节点可用 |
| `position` | 选择岗位 | 按岗位分配审批人 | 按岗位审批 | 不能在起始节点；仅竞签模式 |
| `level` | 选择岗级 | 按职级分配审批人 | 按级别审批 | 不能在起始节点；仅竞签模式 |
| `form_person` | 选择表单人员 | 从表单字段值中提取审批人 | 动态审批人 | 不能在起始节点；仅竞签模式 |

### 4.2 审批方式

| 方式 | type 值 | 说明 |
|------|---------|------|
| 竞签 | `scramble` | 多人中任一人通过即可流转到下一节点 |
| 会签 | `countersign` | 需要指定数量的审批人全部通过后才流转 |

会签配置：
- `countersignNum`：会签所需通过人数（最小值为1）
- 系统通过 `getLastCountersignFlag()` 判断是否为最后一个会签人

### 4.3 各审核类型详细逻辑

#### assign（选择人员）
- 前端通过人员选择对话框选择用户
- 保存到 `FlowNodeAuditDetailConfig` 表，type=personnel
- 运行时直接使用指定的用户列表
- 支持指定多人，根据审批方式（竞签/会签）决定处理逻辑

#### initiator（发起人自己）
- 审批人即为流程发起人
- 前端显示发起人信息
- 运行时从 FlowInstance.initiator 获取用户
- 适用于发起人需要自审的场景

#### run_node_choose（审批人自选）
- 运行时由当前操作人选择下一审批人
- 提交/审批时弹出人员选择对话框
- 选择结果保存到 `FlowNodeAuditDetailConfig`
- 适用于审批人不固定的灵活场景

#### role（选择角色）
- 从 FlowRole 列表选择角色
- 前端显示角色折叠面板及关联用户
- 运行时根据角色查找有权限的用户
- 支持一个节点配置多个角色

#### company_id（选择公司）
- 仅在起始节点可用
- 从公司列表选择
- 运行时根据公司筛选审批人
- 适用于跨公司审批场景

#### position（选择岗位）
- 从岗位列表选择
- 前端通过 JobPosition 组件选择
- 运行时根据岗位匹配用户
- 后端通过 `validateNodeAndSettingsNodeAuditDetailConfig()` 验证

#### level（选择岗级）
- 选择单一职级
- 前端通过 JobTitleSelectDialog 选择
- 运行时按职级匹配用户
- 返回单个职级值

#### form_person（选择表单人员）
- 从表单中选择一个人员类型的字段
- 前端通过 FieldSelectDialog 选择表单字段
- 运行时从 MongoDB 表单数据中提取该字段的值作为审批人
- 支持字段值为 String（单人）或 List（多人）

### 4.4 审核范围（AuditScope）

通过 `FlowNodeAuditScopeService` 管理，限制哪些用户/组可以在某节点进行审批。与审核类型配合使用，进一步缩小审批人范围。

### 4.5 审核配置保存流程

```
saveFlowNodeAuditConfig()
  ├── 1. clearData() — 删除该节点的旧审核配置
  ├── 2. 保存 FlowNodeAuditConfigTemplate（审核类型、审批方式等）
  ├── 3. 保存 FlowNodeAuditDetailConfigTemplate（审核人明细）
  └── 4. 保存 AuditScope（审核范围）
```

### 4.6 运行时审核人校验

`validateNodeAndSettingsNodeAuditDetailConfig()` 方法是核心校验逻辑：

```
输入: FlowNodeProxyVo, FlowInstanceProtocol
  │
  ├── auditType == initiator → 直接通过（发起人节点）
  │
  ├── auditType == run_node_choose → 验证已提供人员列表
  │
  ├── auditType == extendedAttribute → 按扩展属性查询用户
  │     └── 若找到多人但未指定 → 抛出异常
  │
  ├── auditType == form_person → 从 MongoDB 表单数据提取
  │     ├── 字段值为 String → 单人
  │     └── 字段值为 List → 多人
  │
  ├── auditType == level → 按职级查询用户
  │
  └── 其他 → 查询当前处理器用户列表
```

## 5. 条件分支配置

### 5.1 条件节点结构

```
[条件节点]
  ├── 策略1 (FlowStrategy): "金额大于10万"
  │     ├── 条件1: fieldaName="amount", judge=">=", bvalue="100000"
  │     └── 条件2: fieldaName="type", judge="=", bvalue="A", conditionType="AND"
  │     └── 目标节点: [分支A审批人]
  │
  └── 策略2 (FlowStrategy): "默认"
        └── 目标节点: [分支B审批人]
```

### 5.2 条件字段配置

每个条件 (FlowStrategyCondition) 包含：

| 字段 | 说明 |
|------|------|
| fieldaName | 表单字段英文名（左值） |
| bvalue | 固定比较值（右值） |
| fieldbName | 另一个表单字段名（与 bvalue 二选一） |
| judge | 比较运算符 |
| conditionType | 与下一个条件的关系（AND/OR） |
| sort | 排序号 |

### 5.3 比较运算符 (JudgeEnum)

| 运算符 | 说明 | 适用字段类型 |
|--------|------|-------------|
| `>=` | 大于等于 | 数字、日期 |
| `<=` | 小于等于 | 数字、日期 |
| `>` | 大于 | 数字、日期 |
| `<` | 小于 | 数字、日期 |
| `=` | 等于 | 所有类型 |
| `!=` | 不等于 | 所有类型 |
| `IN` | 包含 | 数组、多选 |
| `LIKE` | 模糊匹配 | 字符串 |

### 5.4 条件组合逻辑

- 同一策略下的多个条件通过 `ConditionTypeEnum` 连接：
  - `and` — 所有条件都满足
  - `or` — 任一条件满足
- 条件按 `sort` 排序后依次评估
- 最后一个策略通常作为默认分支（兜底）

### 5.5 手动分支 (custom_choose)

- 节点类型仍为 `condition`，但 `branchExecuteType` 设为 `custom_choose`
- 提交时由用户手动选择走哪个分支
- 前端显示单选按钮列表，用户选择后将选择结果保存到表单数据
- 后端通过 `customChooseConditionBranch()` 解析选择路径

### 5.6 条件评估算法

`FlowNodeProxyServiceImpl.isLastResult()` 方法：

```
输入: 策略列表、表单数据 (formDataMongoVo)
  │
  ├── 遍历每个策略 (FlowStrategyProxy)
  │     ├── 获取该策略的所有条件 (FlowStrategyConditionProxy)
  │     ├── 按 sort 排序
  │     ├── 依次评估每个条件:
  │     │     ├── 获取 fieldaName 对应的表单字段值
  │     │     ├── 获取 bvalue 或 fieldbName 对应的值
  │     │     ├── 根据 judge 运算符比较
  │     │     └── 根据 conditionType 与前一个结果组合 (AND/OR)
  │     └── 返回最终 true/false
  │
  └── 返回第一个结果为 true 的策略对应的目标节点
      └── 若无匹配，返回最后一个策略（默认分支）
```

## 6. 并行节点配置

### 6.1 并行节点结构

```
[并行节点]
  ├── 分支A: [审批人1] → [审批人2]
  ├── 分支B: [审批人3]
  └── 分支C: [审批人4] → [审批人5] → [审批人6]
```

- 所有分支**同时开始执行**
- 每个分支独立流转
- **所有分支都完成后**才继续到下一个主线节点

### 6.2 并行分支同步机制

通过 `FlowParallelLogicRecord` 跟踪：

| 字段 | 说明 |
|------|------|
| flowInstanceProxyId | 并行节点 ID |
| batchNo | 当前批次号 |
| parallelStatus | 分支状态：`unfinished` / `accomplish` |
| branchLineId | 分支线标识 |

**同步流程**：
1. 进入并行节点时，为每个分支创建一条 `FlowParallelLogicRecord`，状态为 `unfinished`
2. 某分支完成时，更新其记录状态为 `accomplish`
3. 检查所有并行分支是否都为 `accomplish`
4. 若全部完成：清除执行上下文，解析下一主线节点，创建待办任务
5. 若未全部完成：继续为剩余分支创建待办任务

### 6.3 空分支跳过

当并行分支内没有配置审批人时：
- `checkPersonnel()` 递归检查分支内所有节点（包括嵌套的并行/条件）
- 若确认无审批人，该分支自动标记为跳过/通过
- 不会为该分支创建待办任务

### 6.4 嵌套场景

系统支持任意深度的嵌套：
- 并行内嵌条件
- 条件内嵌并行
- `reParallelNode()` 递归向上查找最近的并行祖先节点
- `savePassLogicControl()` 管理父级并行等待标志（`isParentParallelWaitFlag`、`isTopParallelWaitFlag`）

## 7. 字段权限配置

### 7.1 字段权限模型

每个审批节点可以配置表单字段的编辑权限：

```javascript
{
  formFieldTemplateId: "字段ID",
  formFieldTemplateEnglishName: "字段英文名",
  fieldPower: "hide",        // 权限类型
  formTemplateId: "表单模板ID"
}
```

### 7.2 权限类型 (FieldPower)

| 权限 | 说明 |
|------|------|
| `hide` | 隐藏字段，当前节点不可见 |
| `only_read` | 只读（默认值，创建表单时所有字段默认 only_read） |
| `edit` | 可编辑（前端通过勾选框设置） |

### 7.3 配置方式

- 前端通过全屏对话框展示表单预览
- 每个字段旁有「可编辑」勾选框
- 勾选 = edit，不勾选 = only_read
- 特殊字段可设为 hide

### 7.4 运行时权限查询

`FlowNodeFieldPowerProxyServiceImpl.getUserHideField()` 方法：
- 输入：nodeProxyId + userId
- 返回：该用户在该节点的隐藏字段列表
- 仅返回 `fieldPower=hide` 的字段

### 7.5 字段权限更新

```
updateNodeFieldPower()
  ├── 1. 删除旧权限记录
  ├── 2. 保存新权限记录
  ├── 3. 重建完整模板树
  └── 4. 通过 editCustomFlowTemplate 重新创建
```

## 8. 流程发起

### 8.1 发起向导（前端）

#### Step 1：选择流程

- 显示可用流程模板列表（单选）
- 支持预览表单内容和流程结构
- 按 `useScope: 'invest'` 和项目 ID 筛选

**API**: `POST /web/schedule/getFlowTemplateList`

#### Step 2：填写表单

## 10. 操作详解

### 10.1 审核（通过/不通过）

**入口**: `FlowAuditServiceImpl.audit()`

**通过 (pass) 流程**:

```
audit(auditStatus=pass)
  ├── 1. 验证待办任务存在且为 pending 状态
  ├── 2. 获取 Redis 锁 (FLOW_AUDIT_LOCK, TTL=3秒)
  │     └── 获取失败 → 返回 "提交中,请30秒后重试"
  ├── 3. 验证流程实例存在且为 run 状态
  ├── 4. 验证审核权限
  ├── 5. 获取当前节点和流程代理
  ├── 6. 判断是否为最后一个会签人 (getLastCountersignFlag)
  │     ├── 非最后一人 → 标记当前任务为 done，等待其他人
  │     └── 最后一人 → 继续通过逻辑
  ├── 7. passLogic() 处理通过逻辑
  │     ├── 获取下一节点（跳过空节点）
  │     ├── 处理并行节点业务逻辑
  │     └── savePassLogicControl() 创建下一节点待办
  ├── 8. 设置表单中的审核信息
  ├── 9. 保存表单数据到 MongoDB
  ├── 10. 更新当前任务状态为 done
  ├── 11. 保存审核记录和附件
  ├── 12. 更新流程实例
  ├── 13. 更新子流程状态（如有）
  ├── 14. 保存业务关联
  ├── 15. 保存下一节点审核人信息
  ├── 16. 发送触发器回调
  ├── 17. 保存运行流程线
  ├── 18. 删除临时表单数据
  ├── 19. 推送消息通知
  │     └── 系统自动审批用户 (ID: "999999999") 有特殊处理
  └── 20. 保存流程跟踪
```

**不通过 (no_pass) 流程**:

```
audit(auditStatus=no_pass)
  ├── 1-6. 同通过流程的校验步骤
  ├── 7. 设置状态为 rejected
  ├── 8. 重置 currentNode 到起始节点
  ├── 9. 删除该节点的其他并行待办任务
  ├── 10. 递增驳回统计次数
  ├── 11-20. 同通过流程的后续步骤
```

**前端审核弹窗 (ExamineDialog)**:

| 区域 | 内容 |
|------|------|
| 左侧 | 表单预览（只读） |
| 右侧上方 | 流程审批记录 |
| 右侧中间 | 附件上传 |
| 右侧下方 | 审批意见（必填，最长200字） |
| 操作按钮 | 通过 / 不通过 / 回退上一节点 |

### 10.2 回退上一节点 (Rollback)

**入口**: `FlowRollBackServiceImpl.rollBack()`

**校验规则**:
1. 流程实例必须为 run 状态
2. 待办任务必须存在且为 pending 状态
3. 任务必须有父任务 (pid 不为空) —— 否则抛出 "当前节点不支持回退"
4. 父任务不能是起始节点 —— 否则抛出 "请直接驳回给发起人"

**回退逻辑**:

```
rollBack()
  ├── 1. 验证实例和任务
  ├── 2. 获取前一个任务和对应节点
  ├── 3. 保存回退审核记录 (ExecuteResultEnum.roll_back_the_previous_level)
  ├── 4. 生成新批次号
  ├── 5. 分支判断：
  │     ├── 前一节点有 strategyId（在分支内）:
  │     │     ├── 查找最近的并行节点
  │     │     ├── 同一并行分支内回退 → 为前一节点创建待办
  │     │     └── 跨分支回退 → 查找所有分支末端，为每个分支创建待办
  │     └── 前一节点是普通/协同节点 → 创建单个待办任务
  ├── 6. 保存批次信息，更新实例
  ├── 7. 标记当前待办为 done
  ├── 8. 推送待办通知
  └── 9. 更新统计
```

**前端按钮**: 审核弹窗和待办列表中显示「回退上一节点」

### 10.3 撤销 (Revocation/Withdraw)

**入口**: `POST /flowInstanceApi/revocation` → Feign → `rsh-cloud-workflow-center-api`

**校验规则**:
1. 流程实例必须存在
2. 流程实例必须为 run 状态
3. 若已撤销 → 抛出 "操作失败,当前流程已撤销"
4. 若已完结 → 抛出 "当前流程已完结"

**效果**:
- 状态变为 `withdraw`
- 记录 `withdrawDesc`（撤销原因）
- 清除所有待办任务
- 前端仅在「已发」Tab 中当 `status == 'run'` 时显示「撤销」按钮

### 10.4 取回 (Retrieve)

**入口**: `FlowInstanceServiceImpl.saveRetrieveProcess()`

**校验规则**:
1. 流程实例必须为 run 状态
2. 不能在起始节点取回
3. 不能重复取回
4. 当前用户必须是审批人
5. 任务必须为 done 状态（或会签节点）
6. 下一节点不能已经被处理

**取回逻辑**:
```
saveRetrieveProcess()
  ├── 1. 全面校验
  ├── 2. 创建新批次
  ├── 3. 创建新的待办任务（给自己）
  ├── 4. 处理并行逻辑
  │     └── 重置分支线为 unfinished
  ├── 5. 处理会签
  │     └── 设置有效性为 false
  ├── 6. 更新所有待办节点的批次号
  └── 7. 保存取回审核记录 (ExecuteResultEnum.retrieve)
```

**前端按钮**: 仅在「已办」Tab 中当 `flowStatus == 'run'` 时显示「取回」

### 10.5 移交 (ApproverAppend/Transfer)

**入口**: `POST /flowInstanceApi/approverAppend`

**说明**: 为当前审批节点添加新的审批人，不改变流程流转方向。

**前端按钮**: 待办列表中的「移交」按钮

### 10.6 转办 (Transpond/Forward)

**入口**: `FlowInstanceServiceImpl.transpond()`

**说明**: 使用"系统默认转办流程"模板创建一个全新的流程实例，与原实例独立。

**前端按钮**: 审核弹窗中的「转办」操作

### 10.7 终止/废弃 (Abandon)

**入口**: `POST /flowInstanceApi/abandon` → Feign → `rsh-cloud-workflow-center-api`

**校验规则**:
1. 流程实例必须存在
2. 流程实例必须为 run 状态

**效果**:
- 状态变为 `termination`（终止）或 `abandon`（废弃）
- 记录 `abandonDesc`（终止原因）
- 清除所有待办任务
- 与 `end` 状态同为终态，只读

### 10.8 催办 (Urge)

**入口**: `UrgeHandleRecordServiceImpl.sendUrgeMessage()`

**逻辑**:
```
sendUrgeMessage()
  ├── 1. 查找流程实例的所有待办任务
  ├── 2. 获取待办用户 ID 列表
  ├── 3. 仅为 TaskStatusEnum.pending 的任务创建催办记录
  ├── 4. 创建 UrgeHandleRecord
  │     ├── type = "1"
  │     ├── status = "0" (未处理)
  │     └── 消息格式: "【{用户名}】催办: {流程名称},需你处理"
  ├── 5. 推送消息通知
  │     └── 消息标题: "催办提醒"
  └── 6. 系统自动催办用户: ID="999999999", 名称="系统自动"
```

**更新催办状态**:
- 读取后将 status 从 `"0"` 更新为 `"1"`

### 10.9 重新发起 (ReSubmit)

**入口**: `FlowReSubmitServiceImpl.reSubmit()`

**校验规则**:
- 仅允许 `rejected`、`withdraw`、`draft` 状态重新发起

**逻辑**:
```
reSubmit()
  ├── 1. 验证实例状态
  ├── 2. 获取起始节点，遍历到首个执行节点
  ├── 3. 清除表单中的旧审核信息 (clearAuditInfoValue)
  ├── 4. 生成新批次号
  ├── 5. 保存表单数据
  ├── 6. 创建首个待办任务
  ├── 7. 保存审核记录
  ├── 8. 保存附件
  ├── 9. 发送触发器回调
  ├── 10. 删除旧运行流程线
  ├── 11. 保存新运行流程线
  ├── 12. 推送待办通知
  └── 13. 设置状态为 run
```

**前端按钮**:
- 待发 Tab: 「重新发起」
- 驳回后: 「重新发起」
- 撤销后: 「重新发起」

## 11. 并行与条件执行逻辑

### 11.1 节点遍历算法

**`getNextNodeProxyIdById()`** — 查找下一节点：

```
输入: 当前节点、表单数据
  │
  ├── 当前节点是 parallel → 返回 null（并行分支由调用方处理）
  │
  ├── 当前节点是 condition → 评估策略条件
  │     ├── 遍历策略列表 (FlowStrategyProxy)
  │     ├── 对每个策略评估其条件列表
  │     ├── 返回第一个匹配策略的目标节点
  │     └── 无匹配时返回最后一个策略（默认分支）
  │
  └── 当前节点是叶节点 (common/synergy)
        ├── 查找子节点 (by parentId)
        ├── 有子节点 → 返回子节点
        └── 无子节点 → 递归向上遍历 (digui)
              ├── 通过 strategyId 查找父策略
              ├── 查找父策略所在条件节点的下一个兄弟策略
              └── 返回下一个兄弟策略的目标节点
```

### 11.2 递归获取下一执行节点

**`recursionGetNextNodeProxyData()`** — 带跳过逻辑的节点查找：

1. 获取下一个节点（通过 `getNextNodeProxyIdById`）
2. 若为 null，尝试查找分支汇合节点 (`cusQueryBranchCommentNode`)
3. 仍为 null，向上遍历并行节点
4. 跳过无审核人配置的节点（`nodeSkip=true` 时）
5. 处理并行到分支的遍历

### 11.3 手动分支处理

**`customChooseConditionBranch()`** — 手动分支选择：

1. 用户在提交/审批时选择分支
2. 选择结果保存到表单数据
3. 后端通过 `customChooseConditionBranch()` 验证选择
4. 检查分支审核人配置
5. 递归处理后续节点

### 11.4 并行完成检测

`savePassLogicControl()` 中的并行完成检测：

```
标志变量:
  topParallelNode       — 最顶层并行节点
  parallelWaitFlag      — 当前分支是否在等待其他并行分支
  parentParallelWaitFlag — 父级并行是否在等待
  topParallelWaitFlag   — 顶层并行是否在等待
  topConditionNode      — 顶层条件节点

检测流程:
  1. 分支完成 → 更新 parallelStatus 为 "accomplish"
  2. 查找所有并行分支记录
  3. 检查是否全部为 "accomplish"
  4. 全部完成 → 清除执行上下文，继续主线
  5. 未全部完成 → 为剩余分支创建待办
```

### 12.1 定时任务类型

| 类型 | type 值 | 说明 |

| 自动审批 | `"0"` | 超时后自动通过审核 |
| 超时提醒 | `"1"` | 超时前发送提醒消息 |

### 12.2 自动审批逻辑

```
asyncCheckNodeInitiateJob()
  ├── 创建 FlowTimerJob
  │     ├── type = "0" (自动审批)
  │     ├── status = 0 (活跃)
  │     └── 设置触发时间
  └── 到达触发时间时：
        ├── 使用 HashedWheelTimer 触发
        ├── 创建审核请求
        │     ├── userId = "999999999" (系统用户)
        │     ├── autoAuditFlag = true
        │     └── executeDesc = "系统自动通过"
        └── 调用 flowAuditService.audit() 执行自动通过
```

### 12.3 超时提醒逻辑

```
触发时间到达前：
  ├── 查找待办用户
  └── 发送提醒消息

定时任务调度：
  ├── FlowAuditScheduler 使用 HashedWheelTimer
  ├── initAllTask() 每分钟执行一次 (cron: 0 0/1 * * * ?)
  └── 启动时重新加载所有未完成/已过期的定时任务
```
