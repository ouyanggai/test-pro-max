 # Python 桌面程序 UI 构建准则

---

## 设计哲学

**功能即界面**。工具软件的 UI 不是装饰，每一个视觉元素都应该承载操作意义。克制优先于丰富，清晰优先于美观。

---

## 布局结构

采用经典三栏布局，从左到右职责递进：左侧窄栏做导航，中央宽区做主操作，右侧面板做上下文详情。三个区域宽度比建议 1 : 3 : 1.5，中央区域永远是视觉重心。

顶部保留一条 32px 工具栏，放置全局操作（运行/暂停/停止），不随内容滚动。底部状态栏 24px，常驻显示当前任务数、全局状态、连接状态，让用户随时感知系统状态而不需要主动去找。

---

## 字体与字号

只用两个字号：14px 做正文和标签，12px 做辅助信息和时间戳。字重只有两档：regular 和 medium，禁止 bold，bold 在工具类软件里视觉噪音太重。

等宽字体（Consolas / JetBrains Mono）只用于接口 URL、请求体、响应体、日志行，其余全部用系统无衬线字体。这个区分非常重要，用户扫视时靠字体就能区分"数据内容"和"界面说明"。

---

## 颜色系统

整体用中性色（浅灰底 `#F5F6F8`，白色卡片 `#FFFFFF`），只有状态信息才引入彩色：

- 成功 / 完成：`#22C55E`（绿）
- 运行中：`#3B82F6`（蓝）
- 暂停：`#F59E0B`（琥珀）
- 失败 / 错误：`#EF4444`（红）
- 不可操作 / 跳过：`#9CA3AF`（灰）

这五种颜色覆盖所有任务状态，不引入第六种。颜色不能是唯一区分手段，每个状态必须同时配图标或文字标签，保证色盲用户也能识别。

---

## 组件规范

**按钮**有明确的三级层级：主操作（实色填充）、次要操作（描边）、危险操作（红色描边）。同一个操作区域内主按钮只出现一个，避免用户犹豫。

**表格和日志列表**是这类工具的核心组件，要求：奇偶行交替底色（差值 4% 亮度即可），鼠标悬停高亮整行，点击选中后右侧面板同步展开该行详情。列宽除时间戳和状态列固定宽度外，其余列允许用户拖拽调整并持久化保存。

**节点卡片**（流程树中的每个操作节点）统一规格：固定宽度 160px，高度 64px，左侧 4px 色条表示状态，卡片内从上到下依次是节点名称、接口路径、耗时。超出宽度的文字用省略号截断，悬停 tooltip 显示完整内容。

**输入框和表单**标签始终在输入框上方，不用 placeholder 做标签替代品，因为用户开始输入后 placeholder 就消失了，字段含义不应该消失。

---

## 交互细节

**键盘优先**。核心操作必须有快捷键：`Space` 全局暂停/恢复，`F5` 运行，`Esc` 停止，`Ctrl+L` 清空日志，`Ctrl+E` 导出报告。快捷键列表在帮助菜单里有完整说明，不依赖用户自己发现。

**操作反馈要即时**。按钮点击后 100ms 内必须有视觉变化（状态改变或 loading 态），超过 500ms 的操作要有进度指示，超过 3 秒的操作要允许取消。用户不应该对着没有反应的界面猜测"到底有没有在执行"。

**破坏性操作要有确认**。停止正在执行的任务、清空日志、删除配置，弹出确认对话框，且对话框聚焦在取消按钮上，而不是确认按钮。默认行为应该是安全的。

**滚动行为要符合预期**。日志列表在有新内容时自动滚动到底部，但如果用户手动向上滚动查看历史，就停止自动滚动，并在底部显示"↓ N 条新日志"的浮动提示，点击后恢复自动滚动。

---

## 状态持久化

用户调整过的列宽、面板宽度、窗口大小和位置，下次启动时恢复。上次执行的配置（并发数、延迟设置、选中的流程）默认保留。这些细节决定了工具是否"懂用户"。

---

## 一条底线

每个功能上线前问一个问题：**一个第一次用这个工具的人，能不能在 30 秒内找到他需要的操作？** 如果不能，界面需要重新梳理，而不是加说明文档。
