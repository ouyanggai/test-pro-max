# 接口编排自动化测试工具：核心技术方案

## 1. 文档定位

本文档描述接口编排自动化测试工具的核心技术设计。

本文档聚焦核心运行能力，不规划 CLI，不把 `DataGetconfig/config` 下的账号和探针流程固化为产品需求。`config` 目录仅作为开发验证环境资料，用于让实现过程可以直接连接真实环境验证。

业务场景和产品需求见《接口编排自动化测试工具_业务场景与产品需求草案.md》。

## 2. 总体架构

推荐架构仍采用 Python 桌面工具路线：

- 桌面框架：PySide6。
- 事件循环：qasync 融合 Qt 事件循环和 asyncio。
- HTTP 客户端：httpx.AsyncClient。
- 本地存储：SQLite。
- 报告输出：HTML、Excel、JSON 优先。

核心模块：

| 模块 | 职责 |
| --- | --- |
| EnvironmentService | 读取环境配置、网关地址、登录路径和非密钥环境资料 |
| SecretProvider | 从本地 ignored env 或系统环境变量读取密码、token 等密钥 |
| UserDirectoryService | 查询用户、部门、岗位，提供选择候选池 |
| SessionManager | 统一登录、SID 缓存、防顶号、SID 自动恢复 |
| FlowCatalogService | 按发起账号查询可发起流程 |
| FlowInspector | 解析流程节点、分支、表单和可选人员/部门字段 |
| RunPlanBuilder | 把 UI 选择转换成内部运行计划 |
| AssignmentEngine | 执行手动、一键、随机、岗位、部门等指派策略 |
| BranchScheduler | 调度同一流程实例内的分支并发执行 |
| NodeExecutor | 执行单个节点动作 |
| RunRecorder | 记录运行计划、节点日志、请求摘要和报告数据 |
| ReportExporter | 导出 HTML、Excel、JSON 报告 |

## 3. 关键设计原则

### 3.1 用户不直接编写 DSL

工具内部需要稳定的运行计划模型，但不要求用户手写 DSL。

用户通过界面完成：

- 选择账号。
- 选择流程。
- 填写表单。
- 按实际审核节点配置完成节点处理设置。
- 点击运行。

系统内部将这些选择保存为 `RunPlan`，用于执行、复现和报告。

### 3.2 流程分支并发，由 SessionManager 统一处理身份

并发单位是流程分支。

分支和节点执行器不能绕过 `SessionManager` 自行管理 SID。节点执行到某个审核动作时，应根据当前节点实际处理人向 `SessionManager` 获取会话。

如果该处理人尚未登录，`SessionManager` 负责自动登录并缓存 SID。如果 SID 失效，`SessionManager` 负责重新登录并返回新的会话，避免因为身份校验问题阻塞流程回归。

### 3.3 环境资料和产品模型隔离

`DataGetconfig/config` 提供开发验证用的真实环境事实，例如网关、Nacos 来源、数据库、Redis、测试租户和本地 env 模板。

技术实现不得把这些资料硬编码到产品模型中，应通过配置服务读取。

## 4. 内部运行计划模型

`RunPlan` 是工具的核心内部契约。它由 UI 配置生成，保存到本地，用于执行和复现。

示例：

```yaml
run_plan_version: 1
name: "付款流程回归-默认场景"

environment:
  env_id: "test"
  gateway_ref: "config.env.GATEWAY_URL"

starter:
  mode: "default_super_account"
  user:
    username: "欧阳改"
    display_name: "欧阳改"

flow:
  flow_id: "flow_xxx"
  flow_name: "付款审批流程"
  select_mode: "starter_visible_flows"

form_data:
  strategy: "fixed"
  fields:
    amount: 12000
    reason: "自动化回归测试"

assignments:
  default_policy: "manual_first"
  random_seed: 20260509
  rules:
    - target:
        node_id: "dept_approve"
        field: "approver"
      selector_type: "user"
      mode: "position_random"
      scope:
        departments: ["财务部"]
        positions: ["部门负责人"]
        exclude_starter: true
        exclude_used: true

branch_execution:
  mode: "parallel_branches"
  max_branch_concurrency: 5
  failure_policy: "wait_all_then_fail"

session_policy:
  sid_reuse: true
  relogin: "only_when_missing_or_expired"
  duplicate_login_guard: true

assertions:
  - type: "workflow_status"
    expect: "completed"
```

## 5. 发起账号与流程目录

### 5.1 默认发起账号

系统内置默认发起账号配置：

```yaml
default_starter:
  username: "欧阳改"
  role: "super_starter"
```

该配置只表示默认选择，不表示硬编码密码或 SID。

密码、验证码、登录密钥等必须从 `SecretProvider` 读取。

### 5.2 指定发起账号

当用户切换发起账号时：

1. `UserDirectoryService` 查询用户。
2. 用户选择账号。
3. `SessionManager` 获取该账号会话。
4. `FlowCatalogService` 使用该账号 SID 查询可发起流程。
5. UI 刷新流程列表。

### 5.3 流程查询契约

`FlowCatalogService` 输出统一结构，屏蔽后端接口差异：

```python
@dataclass
class StartableFlow:
    flow_id: str
    flow_code: str | None
    flow_name: str
    category_name: str | None
    starter_username: str
    raw: dict
```

## 6. 用户、部门、岗位目录服务

`UserDirectoryService` 负责提供人员和组织选择能力。

接口能力：

```python
class UserDirectoryService:
    async def search_users(self, query: UserQuery) -> list[UserBrief]: ...
    async def list_departments(self, query: DepartmentQuery) -> list[DepartmentBrief]: ...
    async def list_positions(self, query: PositionQuery) -> list[PositionBrief]: ...
    async def resolve_candidates(self, scope: CandidateScope) -> CandidatePool: ...
```

核心数据结构：

```python
@dataclass
class UserBrief:
    user_id: str
    username: str
    display_name: str
    department_id: str | None
    department_name: str | None
    position_id: str | None
    position_name: str | None
    enabled: bool

@dataclass
class CandidateScope:
    departments: list[str] = field(default_factory=list)
    positions: list[str] = field(default_factory=list)
    users: list[str] = field(default_factory=list)
    exclude_users: list[str] = field(default_factory=list)
    exclude_starter: bool = False
    exclude_used: bool = False
```

## 7. 会话管理与防顶号

### 7.1 问题

流程分支并发执行时，不同节点可能需要不同处理人执行审核动作。同一个处理人也可能同时出现在多个分支中。

如果每个分支或节点执行器都独立登录，会造成账号互相顶掉、SID 失效、请求失败。正确做法是所有节点执行都向 `SessionManager` 申请处理人会话，由 `SessionManager` 决定复用 SID、首次登录或失效重登。

### 7.2 会话缓存键

`SessionManager` 以以下维度缓存 SID：

```text
env_id + login_type + username
```

示例：

```text
test + user_login + 欧阳改
test + user_login + d4036cc6581141a8b13cf39387c8d6f2_0142
test + manager_login + rshadmin
```

### 7.3 SessionLease

节点执行器拿到的是 `SessionLease`，不是直接管理 SID。

```python
@dataclass
class SessionLease:
    env_id: str
    login_type: str
    username: str
    sid: str
    generation: int
    expires_at: float | None
```

### 7.4 防顶号规则

1. 同一缓存键只允许一个登录动作同时发生。
2. SID 有效时直接复用。
3. 某个节点需要的处理人尚无 SID 时，由 `SessionManager` 自动登录该处理人并写入缓存。
4. 多个分支同时请求同一账号 SID 时，只有第一个请求触发登录，其他请求等待结果。
5. 每次登录成功后，`generation` 自增。
6. 分支请求遇到未登录、SID 失效或账号被顶掉时，先检查是否已有更新 generation。
7. 如果已有新 SID，直接换新 SID 重试。
8. 只有确认当前 SID 真失效，才允许账号级加锁重登。
9. 对提交、审批等有副作用请求，重登后必须先查询节点是否仍处于可处理状态；仍可处理才继续提交，已经处理成功则记录为成功并推进流程。

### 7.5 处理人会话获取流程

节点执行前必须根据节点处理配置解析出当前动作的处理人。

处理流程：

1. `NodeExecutor` 从 `AssignmentResult` 或流程引擎返回的待办信息中确定处理人账号。
2. `NodeExecutor` 调用 `SessionManager.get_session()` 获取该处理人的 `SessionLease`。
3. 如果缓存中没有该处理人的 SID，`SessionManager` 使用 `SecretProvider` 获取登录凭据并自动登录。
4. 登录成功后缓存 SID，并记录一条 `session_log`。
5. 节点请求使用该 `SessionLease` 执行。
6. 请求返回身份失效时，`NodeExecutor` 调用 `SessionManager.refresh_if_needed()`。
7. `SessionManager` 返回新会话后，节点执行器重新查询节点状态，再决定继续提交或记录已完成。

### 7.6 SessionManager 契约

```python
class SessionManager:
    async def get_session(
        self,
        env_id: str,
        login_type: str,
        username: str,
    ) -> SessionLease:
        ...

    async def refresh_if_needed(
        self,
        lease: SessionLease,
        reason: str,
    ) -> SessionLease:
        ...

    async def mark_invalid(
        self,
        lease: SessionLease,
        reason: str,
    ) -> None:
        ...
```

## 8. 流程解析

`FlowInspector` 在用户选择流程后执行。

输入：

- 发起账号会话。
- 流程 ID。
- 环境配置。

输出：

- 起始表单字段。
- 节点列表。
- 分支关系。
- 每个节点可执行动作。
- 每个节点的实际审核配置。
- 每个节点是否需要用户介入配置。
- 每个节点候选人、候选部门、候选岗位、候选角色或动态规则来源。

核心结构：

```python
@dataclass
class FlowNodeDescriptor:
    node_id: str
    node_name: str
    node_type: str
    branch_id: str | None
    operation_types: list[str]
    requires_assignment: bool
    assignment_targets: list[AssignmentTarget]
    action_configs: list[NodeActionConfig]
    raw: dict

@dataclass
class AssignmentTarget:
    field: str
    selector_type: str   # user / department / position / role / expression
    required: bool
    candidate_source: str | None

@dataclass
class NodeActionConfig:
    action_type: str      # approve / reject / countersign / transfer / revoke
    handler_source: str   # assigned_user / candidate_pool / role / department / expression / todo_owner
    requires_login_user: bool
    candidate_source: str | None
    raw: dict
```

`FlowInspector` 只负责把真实流程配置转换成稳定描述，不在解析阶段假定审核节点一定是“选人”或“选部门”。

## 9. 指派引擎

### 9.1 策略类型

`AssignmentEngine` 支持以下策略：

| 策略 | 说明 |
| --- | --- |
| manual | 用户手动指定 |
| one_click | 系统按默认优先级补齐 |
| random | 一键随机 |
| range_random | 用户圈定范围后随机 |
| position_random | 按岗位随机 |
| department_random | 按部门随机 |
| same_as_previous | 沿用前序节点选择结果 |

### 9.2 指派结果

每次指派必须保存完整决策信息：

```python
@dataclass
class AssignmentResult:
    node_id: str
    field: str
    selector_type: str
    mode: str
    selected_id: str
    selected_name: str
    candidate_count: int
    random_seed: int | None
    reason: str
    candidate_snapshot_id: str
```

### 9.3 随机可复现

随机选择必须满足：

- 同一 `random_seed` 和同一候选池可复现相同结果。
- 候选池变化时，报告要能看出变化。
- 每个随机节点保存候选池快照。
- 报告展示最终选中对象和选择原因。

## 10. 分支调度

### 10.1 执行层级

```text
ExecutionRun
  BranchRun
    NodeRun
      RequestLog
```

流程发起后，系统根据流程引擎返回的节点状态识别分支。

### 10.2 并发规则

- 流程发起串行执行。
- 分支内节点串行执行。
- 分支之间并发执行。
- 并发上限由 `max_branch_concurrency` 控制。
- 全局暂停影响所有分支。
- 单分支暂停只影响指定分支。

### 10.3 失败策略

支持三种策略：

| 策略 | 行为 |
| --- | --- |
| stop_failed_branch | 只停止失败分支 |
| stop_all | 任一分支失败后停止全部分支 |
| wait_all_then_fail | 等其他分支结束后整体失败 |

MVP 默认使用 `wait_all_then_fail`，便于尽量收集完整问题信息。

## 11. 节点执行

`NodeExecutor` 负责单个节点动作。

输入：

- `RunPlan`
- `BranchRun`
- `FlowNodeDescriptor`
- `AssignmentResult`

输出：

- 节点状态。
- 请求摘要。
- 响应摘要。
- 业务结果。
- 后续节点线索。

节点执行前必须：

1. 检查暂停状态。
2. 根据节点实际审核配置和指派结果解析当前处理人。
3. 通过 `SessionManager` 获取当前处理人的会话；没有 SID 时自动登录并缓存。
4. 注入节点所需处理配置。
5. 记录请求前快照。

节点执行后必须：

1. 记录响应。
2. 识别业务成功或失败。
3. 更新分支状态。
4. 必要时重新查询流程待办或节点状态。

## 12. 本地存储

SQLite 表建议：

```sql
CREATE TABLE run_plan (
    id TEXT PRIMARY KEY,
    name TEXT,
    version INTEGER,
    env_id TEXT,
    starter_username TEXT,
    flow_id TEXT,
    flow_name TEXT,
    random_seed INTEGER,
    plan_json TEXT NOT NULL,
    created_at REAL
);

CREATE TABLE execution_run (
    id TEXT PRIMARY KEY,
    run_plan_id TEXT,
    flow_instance_id TEXT,
    status TEXT,
    started_at REAL,
    finished_at REAL,
    failure_policy TEXT
);

CREATE TABLE branch_run (
    id TEXT PRIMARY KEY,
    execution_run_id TEXT,
    branch_key TEXT,
    status TEXT,
    started_at REAL,
    finished_at REAL
);

CREATE TABLE node_run (
    id TEXT PRIMARY KEY,
    branch_run_id TEXT,
    node_id TEXT,
    node_name TEXT,
    status TEXT,
    assignee_snapshot TEXT,
    started_at REAL,
    finished_at REAL,
    retry_count INTEGER DEFAULT 0
);

CREATE TABLE request_log (
    id TEXT PRIMARY KEY,
    node_run_id TEXT,
    method TEXT,
    url TEXT,
    request_summary TEXT,
    response_summary TEXT,
    status_code INTEGER,
    duration_ms INTEGER,
    created_at REAL
);

CREATE TABLE session_log (
    id TEXT PRIMARY KEY,
    execution_run_id TEXT,
    env_id TEXT,
    login_type TEXT,
    username TEXT,
    generation INTEGER,
    action TEXT,
    reason TEXT,
    created_at REAL
);

CREATE TABLE candidate_snapshot (
    id TEXT PRIMARY KEY,
    execution_run_id TEXT,
    node_id TEXT,
    field TEXT,
    selector_type TEXT,
    candidates_json TEXT,
    created_at REAL
);
```

请求和响应默认只保存摘要。敏感字段必须脱敏。

## 13. 脱敏策略

以下字段不得原样写入报告或数据库：

- password
- token
- sid
- cookie
- authorization
- client_secret
- 数据库密码
- Nacos token

脱敏规则应在 `RunRecorder` 统一执行，禁止各模块各自处理。

## 14. 报告

MVP 报告优先支持 HTML 和 JSON。

报告内容：

- 运行摘要。
- 发起账号。
- 流程名称。
- 分支执行状态。
- 节点执行状态。
- 节点处理配置和指派结果。
- 随机种子。
- 候选池摘要。
- 会话复用和重登记录。
- 请求/响应摘要。
- 错误原因。

Excel 报告作为第二优先级，PDF 暂不纳入 MVP。

## 15. MVP 技术完成标准

第一阶段完成时，应满足：

1. `EnvironmentService` 能读取开发环境配置。
2. `SecretProvider` 能读取本地密钥，不把密钥写入仓库。
3. `SessionManager` 能按处理人自动登录、缓存 SID、复用同一账号 SID，并在 SID 失效时重登。
4. 默认“欧阳改”能查询可发起流程。
5. 指定账号后能查询该账号可发起流程。
6. `FlowInspector` 能解析流程节点和并行分支。
7. `AssignmentEngine` 能完成手动、一键、随机、岗位、部门指派。
8. `BranchScheduler` 能按分支并发执行。
9. `RunRecorder` 能保存运行计划、分支、节点和会话日志。
10. `ReportExporter` 能生成基础 HTML 或 JSON 报告。
