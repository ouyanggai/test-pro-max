# 接口编排自动化测试工具 PRD v1.0

## 问题陈述

当前流程回归验证主要依赖人工在前端反复切换账号、选择流程、填写表单、配置审核节点、处理待办、观察并行分支和核对结果。这个过程有几个明显问题：

- 流程发起入口不自然。测试人员需要先知道哪个账号能发起哪些流程，再手动登录和切换。
- 审核节点配置复杂。流程节点可能按人员、部门、岗位、角色、发起人、表单字段、审批人自选等多种规则确定处理人，不能简单理解成固定选人。
- 并行分支回归效率低。同一流程实例中多个分支需要并发处理，人工操作慢且容易漏节点。
- 多账号会话容易互相影响。不同分支可能需要不同处理人登录，同一处理人也可能出现在多个分支中，如果重复登录或 SID 失效，会导致账号被顶掉、身份校验失败、流程回归中断。
- 回归结果难复现。随机选择人员、节点处理配置、分支执行路径、请求响应、SID 重登过程如果没有记录，失败后很难复盘。
- 现有环境资料虽然已经能支持真实验证，但缺少一个产品化工具把环境、账号、流程、节点配置和执行报告串起来。

## 解决方案

建设一个面向 RSH Cloud 流程回归验证的桌面工具。用户通过界面完成“选账号、选流程、配置审核节点、运行、看结果”，系统内部负责环境读取、会话管理、流程解析、节点处理配置、分支并发执行和报告记录。

第一阶段聚焦 MVP：

- 默认使用“欧阳改”作为超级发起账号查询可发起流程。
- 支持用户手动切换发起账号，并按该账号权限查询可发起流程。
- 通过接口查询用户、部门、岗位等候选信息，避免手填用户 ID。
- 选择流程后，按真实流程配置解析表单、节点、条件分支、并行分支和审核节点处理规则。
- 支持手动指定、一键指定、一键随机、范围随机、岗位随机、部门随机。
- 同一流程实例内按分支并发执行，每条分支内部按节点顺序执行。
- 所有节点执行通过统一 SessionManager 获取当前处理人会话；处理人未登录时自动登录并缓存 SID，SID 失效时自动重登。
- 保存运行计划、节点处理配置、随机种子、候选池快照、分支执行记录、会话记录、请求响应摘要和最终结果。
- 生成基础 HTML/JSON 报告，支持回归结果复查。

## 实现决策

- 使用桌面工具作为第一阶段交付形态，当前 PRD 不规划 CLI 和 CI。
- 默认发起账号为“欧阳改”，用于查询所有可发起流程；用户可切换其他发起账号并重新拉取流程列表。
- `DataGetconfig/config` 和环境文档只作为开发验证环境资料，不作为产品需求硬编码来源。
- 密钥、密码、验证码、SID、token 只能从本地 ignored env 或系统环境变量读取，不进入文档、报告或仓库。
- 建立 `EnvironmentService` 深模块，统一读取环境、网关、登录路径、租户编码等非密钥配置。
- 建立 `SecretProvider` 深模块，统一读取密码、验证码、登录密钥、Nacos token 等敏感配置。
- 建立 `UserDirectoryService` 深模块，封装用户、部门、岗位、角色等候选对象查询。
- 建立 `SessionManager` 深模块，封装登录、SID 缓存、防顶号、generation、SID 失效重登和会话日志。
- `SessionManager` 的缓存键为环境、登录类型和用户名组合；同一缓存键同时只允许一个登录动作。
- 审核节点执行前，由节点执行器解析当前处理人并向 `SessionManager` 获取会话；处理人无 SID 时自动登录并缓存。
- SID 失效或被顶掉时，由 `SessionManager` 重登；重登后节点执行器必须重新查询节点状态，避免重复提交副作用动作。
- 建立 `FlowCatalogService` 深模块，按发起账号查询可发起流程，屏蔽后端流程查询接口差异。
- 建立 `FlowInspector` 深模块，解析流程表单、节点、条件分支、并行分支、字段权限和真实审核配置。
- `FlowInspector` 不假定审核节点只有选人或选部门，必须支持选择人员、发起人自己、审批人自选、选择角色、选择公司、选择岗位、选择岗级、选择表单人员等配置形态。
- 建立 `RunPlanBuilder`，把用户界面上的账号、流程、表单数据、节点处理配置、随机策略和并发策略转换成内部运行计划。
- 运行计划是内部契约，不要求用户手写 DSL；运行计划必须可保存、可复查、可再次运行。
- 建立 `AssignmentEngine` 深模块，封装手动指定、一键指定、一键随机、范围随机、岗位随机、部门随机、沿用前序选择等策略。
- 随机策略必须保存随机种子、候选池快照、最终选择结果和选择原因。
- 建立 `BranchScheduler` 深模块，调度同一流程实例内的分支并发执行；流程发起串行，分支之间并发，分支内部串行。
- 分支失败策略第一阶段支持只停止失败分支、停止全部分支、等待全部结束后整体失败，默认使用等待全部结束后整体失败。
- 建立 `NodeExecutor` 深模块，执行单个节点动作，负责暂停检查、处理人解析、会话获取、请求执行、状态回查和节点结果输出。
- 节点执行必须遵守流程引擎的真实状态机和操作矩阵，包括运行中、驳回、撤销、终止、完结等状态。
- 第一阶段核心操作以发起流程和审批通过为主；回退、加签、撤销、转办、取回等操作作为后续扩展，但模型要保留扩展空间。
- 建立 `RunRecorder` 深模块，统一保存运行计划、执行实例、分支、节点、请求摘要、会话记录、候选池快照和错误信息。
- 建立 `ReportExporter` 深模块，第一阶段优先输出 HTML 和 JSON 报告，Excel 第二优先级，PDF 不纳入 MVP。
- 报告和本地数据库默认保存请求/响应摘要，敏感字段必须由统一脱敏策略处理。
- 本地存储使用 SQLite，保存运行计划、执行记录、分支记录、节点记录、请求日志、会话日志和候选池快照。
- 流程发起和审核动作应基于真实 API 验证，不以纯 mock 作为完成标准。

## 测试决策

- 好的测试应验证模块对外行为，不绑定内部实现细节。测试应关注输入、输出、状态变化、错误恢复和副作用控制。
- 所有测试用例应保存到项目 `test` 目录并分类放好。
- `EnvironmentService` 需要测试：环境配置读取、缺失配置提示、非密钥配置和密钥引用分离。
- `SecretProvider` 需要测试：本地密钥读取、缺失密钥处理、敏感值不进入日志。
- `UserDirectoryService` 需要测试：按姓名、部门、岗位查询用户，候选池为空、禁用用户、重复用户等边界。
- `SessionManager` 必须重点测试：首次登录、SID 复用、并发请求同账号只登录一次、generation 更新、SID 失效重登、账号被顶后的恢复。
- `FlowCatalogService` 需要测试：默认账号查询流程、切换账号后流程列表变化、无权限账号返回空列表或错误提示。
- `FlowInspector` 必须重点测试：普通审批节点、条件分支、并行节点、嵌套分支、字段权限、八类审核配置解析。
- `AssignmentEngine` 必须重点测试：手动指定、一键指定、范围随机、岗位随机、部门随机、排除发起人、排除已用人员、候选池为空。
- `BranchScheduler` 必须重点测试：分支并发、分支内串行、并发上限、失败策略、暂停恢复。
- `NodeExecutor` 必须重点测试：处理人解析、会话获取、身份失效后重登、节点状态回查、副作用请求避免重复提交。
- `RunRecorder` 需要测试：运行计划保存、节点记录保存、会话日志保存、候选池快照保存、敏感字段脱敏。
- `ReportExporter` 需要测试：HTML/JSON 报告生成、失败节点展示、随机结果展示、会话重登记录展示、敏感字段不泄露。
- 集成测试应至少覆盖默认“欧阳改”查询流程、指定账号查询流程、发起流程、单节点审批、并行分支审批、处理人自动登录、SID 失效重登。
- 测试应尽量使用真实开发环境进行最终验证，但需要有可控开关避免误操作生产或不可逆业务数据。
- 对提交、审批等有副作用接口，测试应先验证节点仍可处理，再提交动作；失败后报告应保留足够信息用于人工清理。
- UI 层测试第一阶段只覆盖核心流程：选择发起账号、选择流程、配置节点、运行、查看报告。

## 不做范围

- 第一阶段不做 CLI 命令行运行。
- 第一阶段不做 CI 集成。
- 第一阶段不做插件系统。
- 第一阶段不做 PDF 报告。
- 第一阶段不做复杂录制回放。
- 第一阶段不做跨环境批量调度。
- 第一阶段不做大规模压测。
- 第一阶段不做流程设计器。
- 第一阶段不完整实现所有流程操作，如回退、加签、撤销、取回、转办、终止；这些只保留模型扩展空间。
- 第一阶段不把账号、密码、SID、token 或 Nacos 访问凭据写入仓库。
- 第一阶段不要求用户手写内部运行计划或 DSL。
